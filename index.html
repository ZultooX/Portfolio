Collision System

I wanted to push my understanding of collisions by creating a custom collision manager, as I felt it would provide deeper insights into optimizing performance and implementing more complex interactions.

Process

After I had a discussion with my group we decided that we would only need AABB (Axis aligned bounding box), which made the development of the system much simpler since I could focus on one shape!

I began the development by exploring different methods of checking collision between AABBs and decided to use regular AABB-AABB detection since we won't be using OBB (Oriented bounding boxes), we would just use axis-aligned.
Since the game was a platformer without depth enabled, it allowed me to implement AABB collision in 2D, which was interesting to me since most of the game is in 3D.

After the initial collision checks were implemented, it was time to figure out how to resolve the collision. I chose to add a force to the collider, which pushed it in the normal direction of the face it collided with, giving it a realistic feel. This introduced a problem where the object would be pushed too far or too little. To fix this, I calculated how far inside the collider the object was and scaled the force with this scalar. This fixed the issue, though now I had to constrain the "Force Engine" so it didn't add force to an axis that was colliding with a collider. To solve this, I divided the physics calculation by each axis and disabled the force on the axis if it collided.

It works!

This shows the first time the collision system works with the "Force Engine". The player can jump move around in a realistic manner.

Problems

Some problems i faced when developing the collision system was mainly the math behind the collision resolution. I tried different ways of blocking the object from moving, until i managed to find a way to calculate the normals of the collision and but also where I should focus on fixing existing bugs or new bugs that appears while my group used the system. I chose to fix the bugs that my group found since, they would have the biggest impact on the actual game itself. Otherwise the bugs wouldn't have been found so fast.

Animation Stack

I wanted to make a way for us to be able to set animations and then that it would go back to the previous animation. So I made a our animations like a stack. The idea was that the it would adapt a Last-In-First-Out principle. That way we could play a walking animation and then transition back to a the idle animation.

Process

To solve this I just made a list which active animations gets added to. And once an animation finishes it pops the back of the vector. The animation player always plays the animation at the back of the vector. But how can I controll how animations are played?

Before you can use animations you have to add the animations to a storage, which loads and stores the animation data. Wich may be "Should I loop", "can i be popped before done playing" and so one. So a walking animation can be pushed to the stack once and keep looping until the MovementController pops it from the stack.

GMTK 2024: Grapple Ascent

You are a mechanicly engineered Yo-Yo built specificly by a foreign civilization to scale an infinietly reaching mountain.

This game was made as part of the annual GMTK game jam. I participated in a group of five people where I mainly focused on the generation of the mountain.

Partially Procedural Mountain Generation

The mmountain only has one model to generate, then i only randomize its rotation. According to some rules set by the biome. The biome can decide what types of obstacles should spawn, the color of the biome, the color of the fog, what deegrees of rotation the mountain should have in different slots. The biomes are randomized using weights incorder to get some control over what biome should be generated instead of relying on pure randomness.
